# v0.1.0 — Project skeleton

> A prompt for the AI. Rules: CLAUDE.md (especially rule 10). Context: docs/proposal.md. Skill: version-workflow.

## Analysis

The first code version. The goal: lay the foundations so that every following version only adds logic — no infrastructure refactors later. What gets built: a strict TypeScript project with the complete tooling (proposal §7), the Fastify app with the adapter architecture (§2), the first endpoint `GET /health` as the proof of the whole pipeline (Zod schema → Swagger → tests → CI), the central error handler with the error model (§3), the local DynamoDB (docker-compose + an idempotent init) and GitHub Actions CI.

The API lives in the repo root until the v0.9.0 monorepo move (§2). Not built now: the Lambda adapter (0.7.0), the `/api/init`, rates, convert and stats endpoints (0.2.0–0.6.0), CORS (with the frontend) and the rate limit (0.11.0) — but the structure expects them.

A note on the error model: the handler returns `{ error: { code, key, message, params? } }` from this version on; the i18n files with the key translations arrive in 0.2.0 — until then the keys and their English messages are defined as constants (the translation-parity test starts in 0.2.0 together with the files).

Open questions: none — two proposal inconsistencies found while writing this prompt were resolved with the human (rule 9): **CI belongs to 0.1.0** (the roadmap wins; §15 said v0.2.0 and was corrected), and **the README stays untouched until 0.8.0** — rule 15's version match (package.json = README badge = CHANGELOG) applies from the version the badges exist.

## Solution design

### Tooling and the project (proposal §7)

- `package.json` — `version: 0.1.0`, `engines: { "node": ">=22" }`; scripts: `dev`, `test:api`, `verify:api` (= typecheck + lint + tests), `lint:api`, `format:api`, `db:init`, `setup`, `openapi:export`; `package-lock.json` committed
- `.nvmrc` (`22`), `.npmrc` (`engine-strict=true`), `.editorconfig` (`indent_style = tab`)
- `tsconfig.json` — strict
- Prettier (`useTabs: true`) + ESLint (`typescript-eslint`, `eslint-config-prettier`; the rules support rule 23 — no `any`, `assertionStyle: 'never'` with the `as const` exception)
- Dependencies — the §11 api column, only what this version uses (rule 8): runtime `fastify`, `zod`, `fastify-type-provider-zod`, `@fastify/swagger`, `@fastify/swagger-ui`, `@fastify/helmet`, `@aws-sdk/client-dynamodb`, `@aws-sdk/lib-dynamodb`; dev `typescript`, `tsx`, `vitest`, `@types/node`, `prettier`, `eslint`, `@eslint/js`, `typescript-eslint`, `eslint-config-prettier`. Deferred with their versions: `@fastify/cors`, `@fastify/rate-limit`, `@fastify/aws-lambda`, `@types/aws-lambda`. Nothing beyond the table.

### The app (proposal §2, §3, §9)

- `src/app.ts` — `buildApp()`: the Zod type provider, `@fastify/helmet`, Swagger UI at `/docs` + the spec at `/docs/json` (generated from the Zod schemas via `jsonSchemaTransform` — no hand-written OpenAPI), request ID (`genReqId` = UUID + the `X-Request-Id` header on every response), the log policy (one line per request: status, latency, request ID; `debug` locally / `info` in production via `NODE_ENV`; never the body; URLs always without the query string), and the **central error handler** — it normalizes ALL errors into `{ error: { code, key, message, params? } }` (404 → `NOT_FOUND`, 413 body over the limit and JSON parse errors → `VALIDATION_ERROR`, anything unexpected → `INTERNAL_ERROR` with no stack trace or internals in the response; the full error goes into the log with the request ID — rule 24)
- `src/server.ts` — the local adapter: listen on `PORT` (default 3000), graceful shutdown (SIGINT/SIGTERM → `app.close()`) (§2)
- `src/schemas.ts` — the Zod schema of the `/health` response + the error response schema (registered in Swagger so the error shape is documented)
- `GET /health` → `{ ok, version, uptime, ratesCacheAge }` (§3) — the version from `package.json`, the process uptime in seconds, `ratesCacheAge: null` (the rates cache exists from 0.3.0); `summary`/`description`/`tags` for Swagger
- `src/lib/` — `constants.ts` (env variable names, defaults, the error keys with their English messages until 0.2.0), `enums.ts` (`ErrorCode`), `types.ts`, `dynamo.ts` (the DocumentClient with the local/AWS switching via `DYNAMO_ENDPOINT` — not used by any route yet, used by `db:init`)
- Every function/method/class with a JSDoc in the rule 12 format; the code entirely in English (rule 21); no `console.log` — only the Fastify logger (rule 7)

### The local environment (proposal §2, §7)

- `docker-compose.yml` (root) — `amazon/dynamodb-local` on port **8002** (8000 is commonly taken) with `user: root` (a named volume) (§2)
- `scripts/create-local-table.ts` — idempotent (`ResourceInUseException` is silently accepted, other errors propagate); `npm run db:init`
- `scripts/setup.ts` — `npm run setup`: `docker compose up -d` + `db:init` + a copy of `.env.example` → `.env` when missing (§7)
- `.env.example` — `PORT`, `DYNAMO_ENDPOINT`, `STATS_TABLE`, without values (the complete configuration list — rule 27); `.env` is loaded by the native `--env-file` in the `dev` script — no dotenv (§7)

### CI and the contract (proposal §7)

- `scripts/openapi-export.ts` → a committed `openapi.json` (the source for the future frontend client and the drift guard)
- `.github/workflows/ci.yml` — the `api` job: `npm ci` → `verify:api` → the **OpenAPI drift guard** (the export + `git diff --exit-code openapi.json`); dynamodb-local as a service container for the integration tests
- `.github/dependabot.yml` — security updates only

### Closing the version (rules 14, 15)

- `CHANGELOG.md` — the `[0.1.0] — YYYY-MM-DD HH:MM` entry (Added)
- The version match: `package.json` = CHANGELOG = 0.1.0

Not done now: `lambda.ts` (0.7.0), `/api/init` and the i18n files (0.2.0), rates/convert/stats (0.3.0–0.6.0), CORS, the rate limit (0.11.0), the README (0.8.0), the frontend (0.9.0+).
