# v0.10.0 — The frontend completion: the statistics, the language changer and the same-origin deploy

> A prompt for the AI. Rules: CLAUDE.md (especially rules 10, 16, 23, 28, 29). Context: docs/proposal.md §8, §9, §10. Skills: version-workflow, fe-component.

## Analysis

Level 2 closes functionally: the three assignment statistics render live in the Result card, the language changer makes the three downloaded languages switchable, the error/loading states get their §10 mapping, the a11y baseline lands — and the whole web deploys to production **same-origin through the Router** (the §10 revision: one CloudFront domain, `/api/*` to the Lambda, everything else to the StaticSite; no CORS exists in production, no chicken-and-egg URL dependency, one URL for the evaluator).

**The v0.2.0 catalog investment pays out the rest of itself:** `ui.numberOfCalculations`, `ui.topTargetCurrency` and `ui.totalAmountEur` have waited in all three languages since the catalog landed complete — the statistics rows need ZERO i18n changes.

**The §10 statistics behavior:** loaded at the boot, refreshed after EVERY successful conversion — the user watches the persistence live. The Figma Result card shows two groups (Result; Number of calculations made) separated by the 1 px divider; the remaining two statistics are the §10 conscious addition — additional rows of the SAME label+value style, the smallest possible intervention.

**One policy pinned here:** a statistics FETCH failure never blocks the converter — the rows render dashes and the next successful conversion retries the refresh; no error state, no banner (the §10 banner belongs to the conversion's own RATE_PROVIDER_ERROR). The §10 empty state (zero conversions: a zero count and a dash for the top currency/total) is a different, honest state — covered by its own test.

**The boot context grows by `languages`** — the changer needs the server list (§3: the frontend hardcodes nothing about languages); a mechanical addition to the ready state (the rule-29 carve-out covers the touched boot test only if an assertion must extend — prefer a new test).

Not built now: the rate limit and the hardening e2e pass (0.11.0); the future vision, the time budget, the license decision (1.0.0).

Open questions: none — §10 decides every behavior; the deploy follows the §8/§10 revised design with the `router.route(prefix, url)` API and the StaticSite attached through `router: { instance: router }`.

## Solution design

### A. The statistics in the Result card (proposal §10; the Figma export)

- `useStats` in `web/src/api/hooks.ts` — the discriminated union + `refresh()`; the generated `getApiStats` through the mutator.
- The Result card gains the §10 rows in the Figma style: the 1 px divider (`rgba(0,0,0,0.2)` — the token exists), then `ui.numberOfCalculations` + `formatCount(totalConversions)`, `ui.topTargetCurrency` + the code (or the dash), `ui.totalAmountEur` + `formatMoney(totalAmountEur, 'EUR')`.
- The wiring: the page fetches the stats once on mount and `refresh()`es after every successful conversion (the success transition of `useConvert` is the trigger — wire it in the page, not inside the hook: the hooks stay single-purpose).
- The pinned failure policy above; the empty state per §10.

### B. The language changer (proposal §10)

- **Centered ABOVE the title with its own spacing — on BOTH breakpoints** (the human's UX decision, §10 amended per rule 9: the originally planned corner placement collides with the title on mobile; one placement for both layouts also removes a responsive special case). The design-system visual (muted text, the field font; no Figma element exists — a §10 conscious addition): the three codes from the BOOT context (`languages` — newly exposed), the active one highlighted.
- A click: `i18next.changeLanguage`, the localStorage `language` write, `<html lang>` update — instant (all the languages are already downloaded; zero requests).
- The codes render as THEMSELVES (EN/CS/SK) — they are not translatable texts, no catalog change.

### C. The §10 error/loading polish

- The Convert button: the in-flight state renders the INLINE spinner inside the button (no skeletons — §10); the button stays the same size (no layout shift). **The in-flight state is announced, not just shown** (the human's review finding — a disabled button with a visual-only spinner is silence for a screen reader): the button carries `aria-busy="true"` while in flight and the inline spinner keeps `role="status"` so assistive tech hears the loading; asserted in the tests.
- **The error mapping is TWO-LEVEL — the code picks the family, the KEY picks the placement** (the human's review finding: `VALIDATION_ERROR` carries the amount keys AND `sameCurrency`/`invalidCurrencyCode`, which belong at the selects — a code-only mapping would park "Source and target currencies must be different" at the amount input; the FE blocks the same-currency pick, but §10 is defense in depth — a backend error must know where to render): `errors.validation.amount*` → the amount field; `errors.validation.{sameCurrency, invalidCurrencyCode}` → the selects; `errors.validation.invalidRequest` → the form-level message; `UNSUPPORTED_CURRENCY` (422) → the selects; `RATE_PROVIDER_ERROR` and the synthetic `NETWORK_ERROR` → the page-level banner. All translated by `key` with `params`; all with `role="alert"`.
- **The catalog gains `errors.network`** (the human-approved recommendation): the v0.9.0 mutator honestly maps transport failures to a key, but `errors.internal` blames the server for the client's network — the new key ("Connection failed — please check your network" + cs/sk) is a DATA addition through `/api/init`, NOT an OpenAPI change (the schema tree is untouched; `openapi.json` stays version-line-only as promised; the ETag changes by design; the parity test enforces all three languages — the exact `ui.retry` precedent). `NETWORK_ERROR_KEY` in the web mutator switches to it.
- **The form defaults (the human's UX decision, §10 amended per rule 9):** the amount pre-fills with `0`; From stays `EUR` and To stays `CZK` (the 0.9.0 defaults, now pinned as design). Two load-bearing details: the pristine `0` is NOT submittable — the validation requires a POSITIVE amount (which also closes a latent 0.9.0 gap: the pattern accepted "0" and the API would have answered 400), so Convert stays disabled until the user types a positive value; and the `0` is FULLY SELECTED on focus, so the first keystroke replaces it (autofocus + `0` without select-all would produce "0100").
- **The title centering fix (the human's report):** on non-desktop devices the H1 renders left-aligned — a 0.9.0 implementation defect, not a design change (§10 has always said "the heading (centered)"). The fix guarantees the centering on EVERY breakpoint and viewport (the element must span the available width with centered text, not rely on its intrinsic size); verified visually in the human sweep on a real non-desktop device.
- The a11y baseline check (§10): `aria-live` on the result and the statistics, the labels wired, the visible focus (`:focus-visible` from the tokens), the combobox ARIA already in place from 0.9.0.

### D. The same-origin production deploy (proposal §8, §10 revised)

- `deploy/sst.config.ts` gains: `sst.aws.Router` with `router.route('/api', api.url)`, `router.route('/docs', api.url)`, `router.route('/health', api.url)` (a prefix matches whole path segments; no rewrite — the Lambda routes carry their prefixes); `sst.aws.StaticSite` with `path: '../web'`, the build command and `output: 'dist'`, attached through `router: { instance: router }` as the default for everything else. The outputs: the single Router URL.
- `VITE_API_URL` stays EMPTY in the production build — the 0.9.0 mutator fallback IS this design working.
- The Function URL keeps `url: true` (the Router points at it); the api-side CORS stays local-only (NODE_ENV guard from 0.9.0) — production never makes a cross-origin request.
- **`/api/stats` and `/health` actively send `Cache-Control: no-store`** (the human's review finding): §3's "no HTTP cache headers" was protection at the direct Function URL and becomes a VACUUM behind the Router — CloudFront may fill a missing Cache-Control with its default TTL, and a cached stats endpoint breaks exactly the refreshed-after-every-conversion behavior (invisibly: the first request is never a hit). The explicit `no-store` defends regardless of the SST cache policy; `/health` gets it as the same class (cached diagnostics lie). §3 is amended per rule 9; the v0.6.0 "NO cache headers" test changes under the rule-29 carve-out — a contract change this prompt explicitly requires, asserting `no-store` instead of absence.
- **The deploy and the verification sweep are the HUMAN's** (deny-listed): `npx sst deploy --stage production`, then over the SINGLE CloudFront URL — the app loads and converts END TO END IN THE BROWSER, `/api/health` + `/api/stats` same-origin, `/docs` + its CSS through the Router, the language switch persists across a reload; the AI prepares the checklist and evaluates.
- The README: the live URL becomes the CloudFront one (the app + the API on one origin); the badge/status maintenance (rule 15).
