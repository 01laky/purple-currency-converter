> A prompt for the AI. Rules: CLAUDE.md (especially rules 10, 24, 29). Context: docs/proposal.md §9, §15. Skills: version-workflow, error-code.

## Analysis

The last version before the submission finalization — three hardening tracks: **the §9 rate limit** (the final piece of the error model: 429 `RATE_LIMITED`), **the adversarial edge-case pass** over the whole system (the /review-api spirit applied repo-wide — find it before the evaluator does), and **the documentation sync** (AI_SETUP.md still shows the v0.0.0-era configuration snapshots; reality moved).

**The rate limit (§9, decided long ago):** `@fastify/rate-limit`, per-IP, the baseline 60 req/min, scoped to `POST /api/convert` ONLY (the §9 letter — the conversion writes statistics, the other endpoints are read-only or cached). The 429 flows through OUR unified error model (`errorResponseBuilder` → `{ code: RATE_LIMITED, key: errors.rateLimited, message }` — the translation has waited in the catalog since v0.2.0, the LAST unused error key comes alive; the catalog investment closes at 100 % utilization). The keying uses the client IP from `x-forwarded-for` — **a HOP COUNT, NOT `true`** (the human's review finding): `true` trusts every hop and yields the LEFTMOST XFF entry — fully client-controlled, so the limiter would be bypassable even THROUGH CloudFront by sending a forged XFF that CloudFront merely appends to; the bounded count resolves request.ip to the RIGHTMOST XFF entry = the viewer IP CloudFront itself appended, leaving client-forged entries ignored. (Implementation note: the review pinned the count as 2 — "the socket is hop 1"; the forged-prefix TEST proved the off-by-one: proxy-addr counts the socket as hop 0, so the correct value is `trustProxy: 1` — trust exactly the one proxy we own. The intent — the viewer IP, never the forged prefix — is unchanged and test-enforced.) A DIRECT Function URL caller can still spoof — accepted per §9, the public entry is CloudFront — and the README states the truth: the limit is abuse DAMPING, not a security boundary. The admitted trade-off goes into the README: the in-memory limit counts PER LAMBDA INSTANCE (like the rates cache) — with N instances the effective limit is N×60; the hard backstop is the account-level concurrency cap of 10 (the v0.7.0 documented decision — the roadmap's "reserved concurrency" resolved there: no per-function reservation is permissible on a quota of 10).

**The frontend needs ZERO code for the 429:** `errorPlacement` already routes every non-validation, non-422 code to the banner, and the catalog carries the text — one additive test case proves it.

**The adversarial pass:** the /review-api command brief applied across `api/` + `web/` — money/rounding, validation edge cases, the outage behavior, untested logic changes — plus the §9 checklist (no secrets in logs, no internals in errors, the headers). The findings are PRESENTED, not auto-fixed (the human's review finding — rule 9): each with options and a recommendation; **the human picks which are fixed in this version**, every accepted fix carries a test, and the rest goes to the Backlog (rule 25). "The AI decides what is real and fixes it" would be a blank check on the scope.

**The documentation sync:** AI_SETUP.md §3 still shows the ORIGINAL settings.json with the dead-feedback-loop hook (`… || true`) that the v0.0.0 audit replaced — the document must show the real exit-2 hook and the grown deny list, with a pointer to the audit story in the diary; the §8 repo tree predates the monorepo move. The README gains the §9 rate-limit trade-off paragraph.

Open questions: none — §9 decides the limit, the scope, the keying and the trade-offs; the two parameters above (the convert-only scope, the bounded trustProxy hop count) are its letter.

## Solution design

### A. The rate limit (§9; the error-code skill — all five places)

- `@fastify/rate-limit` (the §11 table) registered with `global: false`; the per-route enablement goes through the ROUTE OPTIONS — `config: { rateLimit: { max: RATE_LIMIT_MAX, timeWindow: RATE_LIMIT_WINDOW } }` on `POST /api/convert`, NOT through the schema (the human's review finding — the plugin's documented mechanism for the non-global mode); 60 and '1 minute' live in `lib/constants.ts`, no magic values.
- **The max is injectable through the DI seam** (the human's review finding — the test-isolation hazard): the in-memory store lives per app instance, so the cross-FILE isolation is automatic (every test file builds its own app), but within a shared-app file the suite's cumulative converts count into one window — today ~12/60 of headroom in `app.convert.test.ts`, a silent time bomb as tests grow, and rule 29 would forbid the repair. `BuildAppDeps` gains `rateLimitMax?: number` (default `RATE_LIMIT_MAX`); the 429 test runs its OWN app instance with a LOW max (3 → four requests instead of sixty-one, no stats noise — the provider AND the stats repository stubbed), and the existing test apps keep the default with their headroom untouched.
- `errorResponseBuilder` returns the unified §3 shape: `ErrorCode.RATE_LIMITED` + `ErrorKey.RATE_LIMITED` (`errors.rateLimited`) + the EN message via `formatEnglishMessage`; the 429 response schema joins the route's response map — required by rule 6 (the contract lives in `schemas.ts`; an undeclared status would fall back to the default serialization and silently MISS the OpenAPI document), and ONCE declared, the zod serializer validates the builder's output — the builder's shape must match the schema exactly or the serializer turns the 429 into a 500. The catalog test asserts the full body, which proves both.
- `trustProxy: TRUST_PROXY_HOPS` (= 1, corrected from the review's 2 by the forged-prefix test — the socket counts as hop 0) on the Fastify factory — `request.ip` = the viewer IP CloudFront appended (the rightmost XFF entry); client-forged prefixes are ignored. The inject-based tests stay valid: the socket (127.0.0.1) is the trusted hop, a test-supplied XFF header resolves as the client.
- The OpenAPI re-export carries the 429 on `/api/convert`; the web client regenerates.

### B. The adversarial pass (the /review-api brief, repo-wide)

- The AI runs the review with the domain-traps focus (money, validation, outages, the §9 security checklist) over `api/src`, `web/src` and the deploy config; the findings list — each with options and a recommendation — goes into the answer BEFORE any fix, and the work PAUSES for the human's pick (rule 9).
- Each ACCEPTED finding: a failing test first (rule 2 where a bug), then the fix; the declined and out-of-scope ones → the Backlog (rule 25).
- **The explicit edge-case test deliverable** (the human's addition — independent of whether the pass finds bugs): targeted boundary tests for the HELPERS and the LOGIC — primarily the api (money/rounding boundaries, the cache TTL edges, the cross-rate arithmetic, the validation limits, the stats tie-breaks), and on the web the PURE functions only (the formatters, `errorPlacement`, the mutator error mapping, the amount normalization) — NOT the render components, which the existing RTL suites already cover. Additive tests, rule 29 untouched.

### C. The documentation sync (rule 10's honesty applied to docs)

- AI_SETUP.md: the §3 snippet becomes the CURRENT settings.json (the exit-2 hook walking `api/`/`web/`, the deny additions: nested `.env`, `npm run deploy`) with one sentence on the v0.0.0 audit; the §8 tree becomes the real monorepo layout.
- The README: the §9 rate-limit paragraph (the per-instance trade-off, the CloudFront entry, the concurrency backstop); the badge/status maintenance (rule 15).

### D. The production redeploy (the human; deny-listed)

- `npx sst deploy --stage production` after the api change; the live spot check: a normal conversion still 200, and OPTIONALLY the 429 — fired with an INVALID payload (`from = to` → 400), not a valid one (the human's review finding): 60 successful conversions would write 60 records into the production statistics the evaluator sees; the limiter runs in the `onRequest` hook and counts requests BEFORE validation, so 61 invalid requests reach the 429 with ZERO statistics writes.

### Closing the version (rules 14, 15 — in this order)

1. The version 0.11.0 in `api/package.json` FIRST; the README badge/status follow.
2. `api`: `openapi.json` re-exported AFTER the bump (the 429 addition + the version); the web client regenerated.
3. `CHANGELOG.md` — the `[0.11.0] — YYYY-MM-DD HH:MM` entry.
