# v0.2.0 — i18n and GET /api/init

> A prompt for the AI. Rules: CLAUDE.md (especially rule 10). Context: docs/proposal.md. Skills: version-workflow, api-endpoint, error-code.

## Analysis

The texts of the whole system become data served by the backend (proposal §3): the i18n files `src/i18n/{en,cs,sk}.json` (the repo root hosts the API until the v0.9.0 move) carry the backend error messages as well as the future frontend UI texts, and `GET /api/init` serves them all at once with HTTP ETag revalidation. The error handler stops using the temporary `ENGLISH_MESSAGES` constants from v0.1.0 — from now on the `message` field is the EN translation of the key with the params interpolated, so the texts have a single source.

**The key catalog is fixed by §3 and lands complete in this version** — including the keys of error codes that do not exist yet (`errors.rateProvider`, `errors.unsupportedCurrency`, …) and the UI texts. Texts are data, not behavior: defining them now means later versions only add code, and the parity test guards the whole catalog from day one. The `ErrorCode` enum is NOT extended — codes arrive with their versions (0.3.0–0.5.0 errors, 0.11.0 rate limit).

Not built now: `/api/currencies`, rates, conversion (0.3.0–0.5.0), the frontend fallback message (0.9.0), the i18next wiring (0.9.0 — frontend).

Open questions: none — the response shape, the key catalog, the languages, the caching semantics and the parity requirement are all decided in §3.

## Solution design

### The i18n module (proposal §2, §3, §8)

- `src/i18n/{en,cs,sk}.json` — the full fixed catalog (§3): `errors.{notFound, internal, rateProvider, rateLimited, unsupportedCurrency}`, `errors.validation.{invalidRequest, amountNotPositive, amountTooLarge, amountTooManyDecimals, invalidCurrencyCode, sameCurrency}`, `ui.{title, amountToConvert, from, to, convertCurrency, result, numberOfCalculations, topTargetCurrency, totalAmountEur}`. Interpolation placeholders as `{{param}}` (e.g. `"Currency {{code}} is not supported"`). The EN `ui.*` texts come verbatim from the Figma design (§10); CS/SK translations proposed by the AI, reviewed by the human.
- `src/i18n/loader.ts` — **static imports** of the three JSONs (the Lambda bundle has no files on disk — §8): the `languages` list (`['en', 'cs', 'sk']` — ISO 639-1, Czech = `cs`), the translations object, and `formatEnglishMessage(key, params?)` — looks the key up in the EN tree and interpolates `{{param}}`.
- **A missing translation never falls back — it fails immediately (§3).** Under no circumstance does a missing key resolve to the key itself, the English text, an empty string or a placeholder: `formatEnglishMessage` throws on an unknown key (rule 24). The same policy binds the frontend i18n layer at 0.9.0.
- Module-local `types.ts`/`constants.ts` as needed (rule 23): the translation tree type, the language codes.

### The error handler switch (proposal §3)

- The central error handler builds `message` via `formatEnglishMessage(key, params)` — the `ENGLISH_MESSAGES` constant from v0.1.0 is removed.
- Zod schema messages keep carrying keys, never English sentences (§3) — nothing changes in v0.1.0 routes, the rule just gains its real data source.

### `GET /api/init` (proposal §3)

- Response: `{ "languages": ["en", "cs", "sk"], "translations": { "en": {...}, "cs": {...}, "sk": {...} } }` — all languages at once (a few KB, one ETag, instant language switching on the frontend).
- The Zod schema in `schemas.ts`: `languages` as an array of the language codes; `translations` as a per-language map of the recursive string tree (`z.lazy`). Swagger `summary`/`description`/`tags` (the api-endpoint skill chain applies).
- **ETag revalidation:** the ETag = a strong hash (SHA-256, quoted) of the response JSON, computed **once at app build** (the texts are static per process — §3). Every response carries `ETag` + `Cache-Control: no-cache`; a request with a matching `If-None-Match` gets `304 Not Modified` with an empty body. No manual hash endpoint, no custom cache logic — standard HTTP.
