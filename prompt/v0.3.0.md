# v0.3.0 — The rate provider

> A prompt for the AI. Rules: CLAUDE.md (especially rules 1, 3, 10). Context: docs/proposal.md §4. Skill: version-workflow.

## Analysis

The first domain version (Level 1 starts here): the `rates/` module — the openexchangerates client, the generic in-memory cache with the TTL and the stale fallback, and the provider computing every pair through the USD base. **A pure module version: no new endpoint** — `/api/currencies` consumes the module in 0.4.0 and `/api/convert` in 0.5.0. The only visible change is `GET /health`: `ratesCacheAge` gets wired to the real cache (it stays `null` until the first fetch, so the v0.1.0 health test stays green untouched — rule 29).

**Scope boundary for the error model:** the module defines its own error (`RateProviderUnavailableError`) thrown when OER fails and no stale copy exists. The `ErrorCode.RATE_PROVIDER_ERROR` + the central-handler 502 mapping + the catalog API test arrive in **0.4.0** with the first endpoint that can actually trigger them over HTTP — the error catalog is a test (§3) and there is nothing to trigger it through in 0.3.0.

Domain traps in play (CLAUDE.md): the free plan is USD-base ONLY → every pair is a cross-rate through USD, covered by a test; tests never call the real OER API (rule 1) — the fetch is injected everywhere; the external response is always parsed through a Zod schema and the fetch has a timeout (rule 3); the OER `app_id` is a query parameter — URLs are logged without the query string (already enforced by the v0.1.0 log policy).

Not built now: `/api/currencies` (0.4.0 — including the currency-names source reusing the same generic cache), `/api/convert` and `roundMoney` (0.5.0), the 422 supported-currency validation (0.5.0, rule 5), the README trade-off note (0.8.0).

Open questions: none — §4 decides the provider, the cache semantics, the timeout budget and the time handling. One semantic pinned here from §3: `rateTimestamp` = the moment the cached payload was fetched from OER (the cache's `fetchedAt`, ISO 8601 UTC) — with the stale fallback it honestly carries the older time.

## Solution design

### `src/rates/client.ts` — the OER client (proposal §4)

- `fetchLatestRates(deps?)` — GET `https://openexchangerates.org/api/latest.json?app_id=<OER_API_KEY>`; the key from the env (`EnvVar.OER_API_KEY`, missing/empty key = a clear error). The fetch function and the timeout are injectable (`fetchFn` default `globalThis.fetch`, `timeoutMs` default 5000) — rule 1 testability and the §4 timeout budget (5 s < the 10 s Lambda timeout, §8).
- Timeout via `AbortController` — a slow OER never hangs the request.
- The response parsed through a Zod schema (rule 3): `{ rates: Record<string, number>, timestamp: number }` (extra fields ignored). A shape mismatch = an error, never an assumption.

### `src/rates/cached-source.ts` — the generic cache (proposal §4)

- `createCachedSource<T>({ fetchFn, ttlMs, now })` → `{ get(): Promise<CachedValue<T>>, ageSeconds(): number | null }`; `CachedValue<T>` = `{ value: T, fetchedAt: number, stale: boolean }`.
- Fresh within the TTL → served from memory, no fetch. Expired → one fetch; **concurrent callers share the single in-flight fetch** (deduplication — N waiting requests never fire N OER calls).
- **Stale fallback:** a failed refresh with a stale copy in memory → the stale copy with the ORIGINAL `fetchedAt` and `stale: true`; a failed fetch with nothing cached → the error propagates (rule 24).
- **The clock is an injected dependency** (`now(): number`) — the TTL/stale logic is tested deterministically with fake time (§4); no `Date.now()` inside the logic.
- Generic on purpose: 0.4.0 plugs the currency-names source into the same implementation.

### `src/rates/provider.ts` — the cross-rate provider (proposal §4)

- `createRatesProvider(deps?)` wraps the client in a cached source (TTL 10 min) and exposes:
  - `getRate(from, to)` → `{ rate, rateTimestamp }` — **`rate(from→to) = usdRates[to] / usdRates[from]`** (the free plan is USD-base only); the rate in full precision (§3 — no rounding here, ever); `rateTimestamp` = the cache's `fetchedAt` as ISO 8601 UTC.
  - `getSupportedCurrencies()` → the keys of the cached rates (§4 — the source for rule 5 validation at 0.5.0).
  - `getCacheAgeSeconds()` → the cache age for `/health` (`null` before the first fetch).
- A currency missing from the rates throws a module error (`UnknownRateCurrencyError`) — defensive, never a silent `NaN`/`undefined` division; the polite 422 validation happens at the API layer in 0.5.0.
- `RateProviderUnavailableError` — thrown when the fetch fails and no stale copy exists; mapped to 502 in 0.4.0.
- Module-local `types.ts`, `constants.ts` (`OER_BASE_URL`, `RATES_TTL_MS = 600_000`, `FETCH_TIMEOUT_MS = 5_000`), `errors.ts` (rule 23).

### Wiring (proposal §3 /health)

- `buildApp()` creates ONE provider instance; the health handler becomes a factory (`createHealthHandler(provider)`) reading `getCacheAgeSeconds()`. No route fetches rates in this version — `/health` never triggers a fetch.
- `.env.example` gains `OER_API_KEY=` (empty — a secret, rule 27; the README half of rule 27 is deferred with the whole README to 0.8.0).

### Closing the version (rules 14, 15)

- `openapi.json` unchanged (no contract change) — the export run confirms zero drift.
- `CHANGELOG.md` — the `[0.3.0] — YYYY-MM-DD HH:MM` entry; the version 0.3.0 in `package.json`.
