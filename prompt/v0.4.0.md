# v0.4.0 — GET /api/currencies

> A prompt for the AI. Rules: CLAUDE.md (especially rules 1, 3, 10). Context: docs/proposal.md §3, §4. Skills: version-workflow, api-endpoint, error-code.

## Analysis

The first HTTP consumer of the rates module: `GET /api/currencies` returns the codes and names for the frontend selects (§3). The source is OER `/currencies.json`, but **the intersection with the rates is returned** — only currencies that actually have a rate. A currency with a name but no rate would fail conversion with `UNSUPPORTED_CURRENCY`; the intersection means the list and the rates never diverge. The names ride the same generic cache from v0.3.0 (`createCachedSource<T>` — §4 mechanism, built generic for exactly this moment).

**This version completes the deferred error-model piece from v0.3.0:** `ErrorCode.RATE_PROVIDER_ERROR` + the central-handler 502 mapping + the catalog API test — `/api/currencies` is the first endpoint that can trigger it over HTTP. The translation key `errors.rateProvider` has existed since v0.2.0 (the catalog landed complete) — zero i18n churn, exactly as designed.

**A DI seam appears in `buildApp`:** the route tests must not call the real OER (rule 1), so `buildApp(deps?)` accepts an optional `ratesProvider` override; the default stays `createRatesProvider()` and `server.ts` does not change. This seam is the testing surface for every rates-consuming endpoint (0.5.0 reuses it).

Not built now: `/api/convert`, `roundMoney`, the 422 `UNSUPPORTED_CURRENCY` validation (0.5.0); the statistics (0.6.0).

Open questions: none. Two parameters pinned here: the names cache TTL = **1 hour** (matching the §3 `max-age=3600` semantics; names effectively never change — the rates keep their §4-decided 10 minutes), and `/currencies.json` is requested **with the `app_id`** like `/latest.json` (one client pattern, no bet on which OER endpoints tolerate a missing key; the key stays out of logs either way).

## Solution design

### The client extension (proposal §4, rule 3)

- `fetchCurrencyNames(deps?)` in `src/rates/client.ts` — GET `https://openexchangerates.org/api/currencies.json?app_id=<OER_API_KEY>`; the same injectable `fetchFn`/`timeoutMs`, the same AbortController timeout, the same missing-key error. The response parsed through Zod: `z.record(z.string(), z.string())` — codes to display names.
- `CURRENCY_NAMES_URL` and `NAMES_TTL_MS = 3_600_000` in `src/rates/constants.ts`.

### The provider extension (proposal §3, §4)

- A second cached source inside `createRatesProvider`: the names over `NAMES_TTL_MS`, the same injected clock; independent stale fallback (a failed names refresh serves the stale names).
- `getCurrencies()` → `Record<string, string>` — **the intersection**: only codes present in BOTH the names and the rates keys; the values are the display names. Sorted keys (deterministic responses).
- Total unavailability of either source (failed fetch, nothing cached) surfaces as `RateProviderUnavailableError` — same wrapping as the rates.
- The `RatesProvider` type gains `getCurrencies`.

### The error model completion (proposal §3; the error-code skill — all five places)

- `ErrorCode.RATE_PROVIDER_ERROR` in the enum; `ErrorKey.RATE_PROVIDER = 'errors.rateProvider'` (the translations exist since v0.2.0 — the parity test already guards them).
- The central error handler maps `instanceof RateProviderUnavailableError` → 502 with the unified shape (before the generic 4xx/500 branches).
- The 502 response schema registered on the route; the catalog API test actually triggers it over HTTP.

### The route (proposal §3; the api-endpoint skill chain)

- `buildApp(deps?: BuildAppDeps)` — `{ ratesProvider?: RatesProvider }`, default `createRatesProvider()`.
- `GET /api/currencies` — the Zod schema `{ currencies: z.record(z.string(), z.string()) }` in `schemas.ts`; Swagger `summary`/`description`/`tags` (the description states the intersection semantics); response schemas 200 + 502 + 500.
- The response carries **`Cache-Control: public, max-age=3600`** (§3) — the browser/CDN side; the server side is the provider's cache.

### Closing the version (rules 14, 15 — in this order)

1. The version 0.4.0 in `package.json` FIRST.
2. `openapi.json` re-exported AFTER the bump (the document embeds `info.version` — the v0.3.0 CI lesson).
3. `CHANGELOG.md` — the `[0.4.0] — YYYY-MM-DD HH:MM` entry.
