# v0.5.0 — POST /api/convert

> A prompt for the AI. Rules: CLAUDE.md (especially rules 1, 4, 5, 10). Context: docs/proposal.md §3, §5. Skills: version-workflow, api-endpoint, error-code.

## Analysis

The heart of the assignment: the conversion endpoint. Three pieces — `roundMoney()` (the ONLY place money is rounded, rule 4), the conversion service (`src/conversion/`) and the route with the full §3 validation. Statistics are NOT written yet (0.6.0) — the conversion is pure computation in this version.

**The money trap (§5, CLAUDE.md domain traps):** a naive `Math.round(x * 100) / 100` fails on the float representation — `1.005 × 100 = 100.4999…` rounds DOWN. `roundMoney` therefore performs half-up on an **integer representation derived from the decimal string** of the value: the cents come from string manipulation + BigInt, never from multiplication. The boundary tests (1.005, 0.005, 2.675, large values, long expansions) are the proof.

**The error model grows by its last v1-scope piece:** 422 `UNSUPPORTED_CURRENCY` with `params.code` (rule 5 — validated against the ACTUAL supported list, the keys of the cached rates). And the central handler finally honors §3 fully: **Zod messages carry the i18n key directly** — the handler extracts the key from the Zod issue and passes it through into `key` (until now every 4xx collapsed into the generic `errors.validation.invalidRequest`; the specific keys have been waiting in the catalog since v0.2.0).

Not built now: the statistics write (0.6.0 — `TransactWriteItems`, the EUR conversion at write time), the rate limit (0.11.0), the frontend.

Open questions: ONE was raised and **resolved with the human** — may `result` round to 0.00? The minimum valid amount is effectively 0.01 (positive, max 2 decimals); 0.01 converted into a much weaker target currency can half-up-round to 0.00 (e.g. 0.01 JPY → EUR ≈ 0.00006 → 0.00). The proposal is silent. **The human's decision: keep the honest zero** — `result: 0` is the correct mathematical answer; `rate` is returned in full precision so the response stays verifiable, the assignment defines no minimum result, and a floor would be an invented business rule. Covered by an explicit test.

## Solution design

### `src/lib/money.ts` — `roundMoney()` (proposal §5, rule 4)

- `roundMoney(value: number): number` — **half-up to 2 decimal places** (0.005 → 0.01), the single rounding place of the whole system.
- The algorithm shifts the decimal point **through the decimal string**: take `value.toString()`, split the integer and fraction parts, build the cents as `BigInt` from the first two fraction digits, and when the third fraction digit is ≥ 5, add one cent. No float multiplication anywhere on the path. Exponential notation and non-finite inputs throw (the validation bounds make them unreachable; rule 24 — loud, not silent).
- **The integer edge case is explicit:** `value.toString()` of a whole number has NO decimal point (`100` → `"100"`, never `"100.00"`) — the split must treat a missing fraction part as `""` and the result passes through unchanged. Covered by a test.
- Defined for the non-negative inputs the validation guarantees.

### `src/conversion/` — the service (proposal §2, §5)

- `service.ts` — `convertAmount(input, ratesProvider)`: ① validate `from`/`to` against `getSupportedCurrencies()` (rule 5) — an unknown code throws `UnsupportedCurrencyError(code)`; ② `getRate(from, to)` — the cross-rate in full precision; ③ `result = roundMoney(amount × rate)` — the computation in a number, ONE rounding at the end (§5). Returns `{ amount, from, to, rate, result, rateTimestamp }`.
- `errors.ts` — `UnsupportedCurrencyError` with the `code` (the §2 structure); local `types.ts`.
- **`UnsupportedCurrencyError` vs `UnknownRateCurrencyError`:** the service-level 422 check (①) runs BEFORE `getRate`, and the supported list and the rates are the SAME cached payload — so the provider's defensive `UnknownRateCurrencyError` is unreachable for user input by design. If it ever escapes anyway, it correctly surfaces as 500 (an internal inconsistency), NOT as 422 — the two errors answer different questions ("is this currency offered?" vs "did the rates map lose a key mid-flight?").

### The contract (proposal §3; the api-endpoint skill chain)

- `convertRequestSchema` in `schemas.ts` — every message is an i18n KEY: `amount` = `z.number()` (NaN/Infinity rejected) `.positive()` (`errors.validation.amountNotPositive`) `.max(1e12)` (`errors.validation.amountTooLarge`) + a refine for max 2 decimal places (`errors.validation.amountTooManyDecimals`) — **the refine checks the DECIMAL STRING of the value** (the fraction-part length of `toString()`), never `value * 100 % 1` float arithmetic (the same §5 philosophy as roundMoney); `from`/`to` = **`z.string().regex(/^[A-Za-z]{3}$/)`** (`errors.validation.invalidCurrencyCode`) — the regex, not a mere length check (a 3-char string of digits or symbols is not a currency code), normalized to UPPERCASE by the schema transform; a body-level refine `from !== to` AFTER the normalization — `eur`/`EUR` is the same currency (`errors.validation.sameCurrency`).
- `convertResponseSchema` — **every field explicitly typed**: `amount: z.number()`, `from: z.string()`, `to: z.string()`, `rate: z.number()` in full precision (not money — §3), `result: z.number()` rounded (the only rounded field), `rateTimestamp: z.iso.datetime()` (the Zod 4 form of `z.string().datetime()` — the contract guarantees a valid ISO 8601 UTC timestamp) with the §3 semantics IN THE DESCRIPTION (the time the rates were fetched from OER, not the conversion moment; honestly older under the stale fallback).
- The route registers the 200/400/422/502/500 response schemas; Swagger `summary`/`description`/`tags: ['conversion']`.

### The central handler extension (proposal §3; the error-code skill — all five places)

- **The Zod-key passthrough:** a schema validation failure is detected with **`hasZodFastifySchemaValidationErrors(error)`** (from `fastify-type-provider-zod`) and the key is read from **`error.validation[0].message`** — the message of the FIRST Zod issue IS the i18n key (§3) → 400 `VALIDATION_ERROR` with that key; the generic `errors.validation.invalidRequest` stays only for non-Zod 4xx internals (JSON parse, body limit).
- `ErrorCode.UNSUPPORTED_CURRENCY` + `ErrorKey.UNSUPPORTED_CURRENCY` (+ the five validation `ErrorKey`s used by the schemas); `instanceof UnsupportedCurrencyError` → 422 with `params: { code }` — the message interpolates to "Currency XYZ is not supported" (the translation exists since v0.2.0).
