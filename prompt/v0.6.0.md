# v0.6.0 — Statistics: DynamoDB counters and GET /api/stats

> A prompt for the AI. Rules: CLAUDE.md (especially rules 1, 10). Context: docs/proposal.md §3, §5, §6. Skills: version-workflow, api-endpoint, dynamo-stats.

## Analysis

The persistence version — Level 1 closes and the L2 persistence lands with it: every successful conversion writes atomic counters into DynamoDB and `GET /api/stats` reads them back. In-memory state does not survive on Lambda (the CLAUDE.md domain trap) — the statistics go ALWAYS to DynamoDB, surviving restarts and shared across clients.

**The §6 data model is immutable:** one table, `pk = STATS`; `sk = GLOBAL` carries `conversionCount` and `totalEurCents`; `sk = TARGET#<CUR>` carries the per-target `count`. The write is **one `TransactWriteItems` with two `Update ADD`s** (the global counters + the target counter) — both succeed or neither does, `conversionCount` can never drift from the sum of the target counters; no read before write. The read is **one Query `pk = STATS`** — the global values plus the maximum of the `TARGET#` items; **the tie-break is the alphabetically first currency** (determinism = testability).

**Money discipline carries over (§5):** the aggregate is `totalEurCents` — an INTEGER; summation accumulates no float error. Every conversion is converted to EUR at write time with the rate valid at that moment: `amountEur = roundMoney(amount × rate(from→EUR))`. The cents come from the 2-dp-rounded value via `Math.round(value × 100)` — the half-up already happened in `roundMoney`, the `Math.round` only absorbs the float representation error (≤ 1e-9 on a 2-dp value), which is orders of magnitude below the half-cent threshold. **The `from = EUR` branch returns CENTS too** — §5 says "simply the amount", but that sentence speaks about `amountEur` (euros); `toEurCents` returns the integer: `Math.round(amount × 100)` directly, no rate lookup — `100.50` EUR is `10050` cents, never `100.5`. Covered by tests on both branches.

**The write-failure policy (§6):** a conversion NEVER fails because of statistics — the user always gets the result. The transaction retries ×3 with a short backoff (the delay injectable for tests); once exhausted, the error is logged with the request id and the response still goes out as 200. **The write is awaited, not fire-and-forget** — pinned here: Lambda freezes the process right after the response (§8), a background write would silently die; three quick attempts cost milliseconds and are the only reliable option.

**One contract detail pinned here:** `topTargetCurrency` is **nullable** — `null` at zero conversions. The §3 example shows a populated state; §10 explicitly designs the frontend empty state at 0 conversions, so the API must be able to express it honestly (no fake `"EUR"`, no empty string — the no-fallback spirit).

Not built now: the SST/Lambda wiring (0.7.0 — the table `link`), the FE statistics display (0.10.0), the event log (Backlog, §6).

Open questions: none — §6 decides the model, the transaction, the retry and the tie-break; §5 the cents and the EUR conversion; the two parameters above are pinned with their reasoning.

## Solution design

### `src/stats/` — the module (proposal §2, §6; the dynamo-stats skill)

- `repository.ts` — `createStatsRepository(deps?)`: the DocumentClient from `lib/dynamo.ts` (the single access point), the table name from `EnvVar.STATS_TABLE` (the local default `DEFAULT_STATS_TABLE`); deps allow a client/table/delay override for tests.
  - `recordConversion({ targetCurrency, amountEurCents }, log)` — ONE `TransactWriteItems`: `Update GLOBAL ADD conversionCount 1, totalEurCents :cents` + `Update TARGET#<CUR> ADD count 1`. Retries ×3 with the injectable backoff. **The logging is two-level:** the handler passes its REQUEST-SCOPED logger in (the repository has none of its own — this is how the retry lines carry the request id); each transient attempt failure logs **`warn`** (`debug` would be invisible in production where the level is `info` — §9, and a flapping DynamoDB is exactly what you want to see) with the attempt number; the final failure after 3 attempts **throws**, and the handler's catch logs **`error`** with `{ err }` via `request.log` (the request id is bound automatically) while the 200 still goes out.
  - `getStats()` — one `Query pk = STATS`; `totalAmountEur = totalEurCents / 100`; the empty table → zeros + `topTargetCurrency: null`.
  - **The tie-break is free:** the Query returns items in ascending sort-key order (`ScanIndexForward` defaults to `true`) — `GLOBAL` first (`G` < `T`), then `TARGET#` items in alphabetical currency order. One pass with the STRICT comparison `count > currentMax` therefore yields the alphabetically-first winner on ties automatically — **no `.sort()`, and never `>=`** (that would keep the LAST tied currency, not the first). The reliance on the default ascending order is stated in a code comment and guarded by the tie-break test; pagination is a non-issue (~170 target items, far under the 1 MB Query limit).
- `eur.ts` — `toEurCents(amount, from, ratesProvider)` returns an INTEGER (cents): `from === 'EUR'` → **`Math.round(amount × 100)` directly (no rate lookup)** — never the raw amount; otherwise `Math.round(roundMoney(amount × getRate(from, 'EUR').rate) × 100)` (the same cached payload as the conversion — no extra OER call).
- Local `types.ts` and `constants.ts` (`STATS_PK`, `GLOBAL_SK`, `TARGET_SK_PREFIX`, the retry attempts/delay) — rule 23.

### The wiring (proposal §3, §6)

- `BuildAppDeps` gains `statsRepository?` — the DI seam pattern of v0.4.0.
- The convert handler: after a successful `convertAmount`, the WHOLE statistics step runs inside **ONE try/catch wrapping BOTH `toEurCents` AND `recordConversion`** — `toEurCents` calls `ratesProvider.getRate(from, 'EUR')` and can itself throw (`UnknownRateCurrencyError` on a payload missing EUR, or any provider failure); wrapping only the write would let the EUR-conversion leg propagate into the central handler as a 502/500 instead of the guaranteed 200. **Any statistics-step error is caught, logged with the request id and the 200 response goes out unchanged** (§6 — the conversion result is never held hostage by the statistics). The rates needed for the EUR conversion come from the same provider instance.
- `GET /api/stats` — the schema in `schemas.ts` with **explicit Zod types**: `totalConversions: z.number().int().nonnegative()` (an integer counter — OpenAPI says `integer`, not `number`), `totalAmountEur: z.number().nonnegative()` (cents / 100 — never negative; the serializer enforces the invariant loudly), `topTargetCurrency: z.string().nullable()`; **no HTTP cache headers of any kind** (§3 — the statistics are always fresh); Swagger `summary`/`description`/`tags: ['statistics']`; the response schemas 200/500.
