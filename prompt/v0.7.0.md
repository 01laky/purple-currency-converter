## Analysis

The first infrastructure version (the "bonus" IaC of the assignment ‚Äî a full part of this design): `deploy/` with the SST v4 config, the thin Lambda adapter, and the two esbuild traps ¬ß8 solves by design. **No contract change** ‚Äî `openapi.json` must come out of the export byte-identical. The production deploy and the live URL are 0.8.0; this version ends with the stack verified through `sst dev`.

**The adapter stays blind (¬ß2):** `src/lambda.ts` is the AWS twin of `server.ts` ‚Äî `@fastify/aws-lambda` around `buildApp()`, nothing else. The app keeps reading configuration EXCLUSIVELY from env variables; the SST `link` exists for IAM (the function may touch exactly its one table ‚Äî ¬ß9 least privilege), while `STATS_TABLE` and `OER_API_KEY` arrive as plain env vars. No SST SDK import anywhere in `src/` ‚Äî the app must not know it is hosted.

**The two ¬ß8 esbuild traps:** ‚ëÝ the i18n JSONs are statically imported ‚Äî solved since v0.2.0, nothing to do; ‚ë° the Swagger UI assets do not exist in the Lambda bundle ‚Äî `copyFiles` ships `@fastify/swagger-ui/static` next to the bundle as `static/`, and `lib/swagger.ts` (`resolveSwaggerStaticDir()`) hands the plugin an explicit `baseDir` when that directory exists next to the module (on Lambda), falling back to the default `node_modules` resolution locally.

**Known risk, pinned here:** `deploy/` is its own npm project (the ¬ß2 no-sharing principle ‚Äî its own `package.json` with the single `sst` dependency) and the repo root hosts the API until the v0.9.0 move, so `sst.config.ts` references the handler as `../src/lambda.handler` ‚Äî a path OUTSIDE the SST root. esbuild follows relative paths without complaint, but if SST itself refuses an out-of-root handler, the fallback is moving `sst.config.ts` to the repo root for v0.7.0‚Äì0.8.0 and into `deploy/` with the v0.9.0 `git mv` ‚Äî to be resolved with the human only if the risk materializes.

**One parameter pinned here:** the region is **eu-central-1** (Frankfurt ‚Äî the EU home region; the proposal does not name one). Confirm or override with the prompt review.

Not built now: the production stage deploy + the live URL + the README (0.8.0), `sst.aws.StaticSite` and the Router (0.10.0 ‚Äî the frontend), the rate limit (0.11.0).

Open questions: none beyond the two pinned items above ‚Äî ¬ß8 decides the Lambda parameters, the stages, the secrets and the bundling; ¬ß9 the concurrency cap.

## Solution design

### `deploy/` ‚Äî the SST v4 project (proposal ¬ß2, ¬ß8)

- `deploy/package.json` ‚Äî the single dev dependency `sst`; its own `node_modules`; the SST commands run FROM `deploy/` (`npx sst dev`, `npx sst deploy` ‚Äî the latter human-only, deny-listed).
- `deploy/sst.config.ts`:
  - **The MANDATORY first line:** `/// <reference path="./.sst/platform/config.d.ts" />` ‚Äî SST v4 generates the type definitions for the `$config`/`$app`/`sst.*` globals into `.sst/platform/config.d.ts`; without the triple-slash reference the config does not typecheck at all. The referenced file is GENERATED (`.sst/` is gitignored) ‚Äî it exists only after the first SST run from `deploy/`. This line is also the concrete target of the lint exception below (`@typescript-eslint/triple-slash-reference` forbids exactly this).
  - `sst.aws.Dynamo` `Stats` ‚Äî `pk`/`sk` string keys (the ¬ß6 table), on-demand billing; **every stage gets its own stack including its own table** ‚Äî dev conversions never pollute production statistics (¬ß8).
  - `sst.Secret` `OerApiKey` ‚Äî the key lives in SST secrets, never in the repo (¬ß9); set per stage by the human (`npx sst secret set OerApiKey <value>`).
  - `sst.aws.Function` with `url: true`, `link: [stats, oerApiKey]` (IAM ‚Äî exactly its one table), and the env the app already understands: `STATS_TABLE` (the table name), `OER_API_KEY` (the secret value), `NODE_ENV: 'production'` (drives the `info` log level ‚Äî ¬ß9). `DYNAMO_ENDPOINT` stays unset ‚Äî the empty value IS the AWS mode of `lib/dynamo.ts`.
  - **The Lambda parameters are a documented decision, not defaults (¬ß8)** ‚Äî in the config WITH the why-comments: `runtime: 'nodejs22.x'` (matches the pinned local Node), `memory: '512 MB'` (CPU scales with memory ‚Üí faster cold starts), `timeout: '10 seconds'` (above the 5 s OER fetch budget ‚Äî a slow OER ends as a controlled stale/502, never as Lambda killing the request), the concurrency cap of 10 via the ACCOUNT-LEVEL quota (no per-function `reserved` ‚Äî with a quota of 10 AWS leaves no room for reservations, ¬ß9).
  - `copyFiles: [{ from: '../node_modules/@fastify/swagger-ui/static', to: 'static' }]` ‚Äî the ¬ß8 Swagger trap. **The `../` is load-bearing:** `copyFiles.from` resolves relative to `sst.config.ts` (= `deploy/`), and `deploy/node_modules` contains ONLY `sst` ‚Äî `@fastify/swagger-ui` lives in the ROOT `node_modules`. Without the `../` the copy targets a non-existent directory and Swagger UI 404s on Lambda. (At the v0.9.0 move the path shifts again to `../api/node_modules/...` ‚Äî part of the move checklist.)
- The root `.gitignore` already covers `.sst/`.
- The lint setup gets a scoped exception for `deploy/sst.config.ts` ‚Äî the mandatory triple-slash reference above violates `@typescript-eslint/triple-slash-reference`; the exception is file-scoped and documented inline (rule 23's SAFETY spirit).

### The adapter and the Swagger resolver (proposal ¬ß2, ¬ß8)

- `src/lambda.ts` ‚Äî `awsLambdaFastify(await buildApp())` exported as `handler`; a thin adapter with the rule-12 JSDoc and nothing else (the ¬ß2 twin of `server.ts`).
- `src/lib/swagger.ts` ‚Äî `resolveSwaggerStaticDir()`: an existing `static/` directory next to the module ‚Üí its absolute path (the Lambda bundle); otherwise `undefined` (locally the plugin defaults to `node_modules`). The existence check is injectable for tests.
- **"Next to the module" means `import.meta.url`, explicitly:** the project is ESM (`"type": "module"`) ‚Äî `__dirname` DOES NOT EXIST and `process.cwd()` is wrong (it tracks the working directory, not the file). The pattern is `path.join(path.dirname(fileURLToPath(import.meta.url)), 'static')`. It works on Lambda precisely because SST bundles the function as ESM ‚Äî `import.meta.url` points at the bundle file, and `copyFiles` places `static/` right next to it.
- `src/app.ts` ‚Äî the swagger-ui registration gains `baseDir: resolveSwaggerStaticDir()`.
- New dependencies per ¬ß11: runtime `@fastify/aws-lambda`; dev `@types/aws-lambda` (+ `sst` inside `deploy/`). Nothing else (rule 8).
