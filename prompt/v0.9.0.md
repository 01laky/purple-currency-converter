## Analysis

The biggest version of the roadmap — Level 2 starts. Two halves: **the monorepo move** (the §2 target structure becomes reality) and **the frontend base** (Vite/React per §10, pixel-perfect to the committed Figma export). The backend and the frontend share NO code — the only artifact between them is `openapi.json`, from which the frontend GENERATES its client (§2/§10).

**The move (`git mv`, history preserved):** everything API-owned goes from the root into `api/` — `src/`, `tests/`, `scripts/`, `package.json` + lockfile, `tsconfig.json`, `eslint.config.js`, `.prettierrc`/`.prettierignore`, `vitest.config.ts`, `.env.example`, `openapi.json`. The root keeps the repo-wide artifacts: `docker-compose.yml`, `.github/`, `deploy/`, `docs/`, `prompt/`, the four root MD files, `.nvmrc`/`.npmrc`/`.editorconfig`/`.gitignore`. **The pinned follow-ups from earlier versions land here by name:** ① `deploy/sst.config.ts` — the handler becomes `../api/src/lambda.handler` and the copyFiles path `../api/node_modules/@fastify/swagger-ui/static` (the v0.7.0 review); ② the CI api job gains `working-directory: api` (and its drift guard runs there); ③ the PostToolUse hook already walks `api/`/`web/` tsconfigs — the v0.0.0 audit fix waits for exactly this moment, NOTHING to change; ④ the README Quick Start paths gain `cd api`/`cd web` (rule 19 — the README change belongs to this version). The root `package.json` disappears — the commands are `cd api && npm run …` exactly as CLAUDE.md documented from day one. Mechanical test-import updates are the rule-29 carve-out, covered by this prompt.

**The scope split 0.9.0 vs 0.10.0 (pinned):** 0.9.0 = the app boots (init + currencies), the conversion form works end to end (validate → POST → the Result value displays), BOTH layouts (web + mobile, the 920 px grid switch), the custom ARIA-combobox selects (§10 binds their behavior — they ARE the form), `formatMoney` + the helpers. 0.10.0 = the statistics rows in the Result card, the language changer, the full error/loading-state polish, the a11y pass, the StaticSite/Router deploy. The "Number of calculations made" element from Figma arrives with the statistics in 0.10.0.

**The design tokens are SUPPLIED, not estimated** (the figma-to-scss skill): the committed Figma CSS export in **`docs/figma/style`** (Web + Mobile sections) is the binding source — every hex, font size, dimension, spacing and radius comes from there into `web/src/styles/_tokens.scss`; nothing is eyeballed from the screenshot (`docs/figma/Screenshot…png` serves as the visual reference). The breakpoint is **920 px** (§10 — mobile ≤ 919). Roboto is self-hosted via `@fontsource/roboto`.

**The no-fallback policy reaches the frontend (§3, the v0.2.0 decision)** — the EXACT i18next configuration is pinned (the human's review demanded precision and the precision surfaced the real gap): **`fallbackLng: false`** (the default `dev` fallback would silently serve the English text for a key missing in cs/sk — LITERALLY the forbidden fallback, and the missing-key handler would never even fire), **`parseMissingKeyHandler: (key) => { throw new Error(…) }`** (the documented init option for missing-key resolution — a throw propagates through `t()`; no `saveMissing` machinery needed, that exists for reporting keys to backends), and **`returnEmptyString: false`** (an empty string counts as missing → the handler → the throw, per the policy). The single hardcoded string in the whole app is the boot-failure message ("Failed to load application").

Not built now: the statistics display, the language changer, the FE deploy (0.10.0); the rate limit (0.11.0). CORS lands NOW — `web:5173` → `api:3000` is cross-origin locally (§10): `@fastify/cors` with `FRONTEND_ORIGIN` (default `http://localhost:5173`), an exact origin, never `*`; production stays same-origin via the 0.10.0 Router, so the Function URL cors option stays off.

Open questions: none — §10 decides the architecture, the §11 table the dependencies, the scope split and the token source are pinned above.

## Solution design

### A. The monorepo move (proposal §2)

- `git mv` per the list above; `api/package.json` keeps ALL existing scripts unchanged; the api job in `.github/workflows/ci.yml` gains `defaults.run.working-directory: api`; the OpenAPI drift guard compares `api/openapi.json`.
- `deploy/sst.config.ts`: the two `../api/` path updates (the handler + copyFiles) with the comments adjusted.
- Mechanical updates only where paths force them (rule 29's carve-out): nothing else in the api code changes; `cd api && npm run verify:api` must stay green BEFORE the web work starts — the move is its own verifiable step (and its own commit).

### B. The web scaffold (proposal §7, §10, §11)

- `web/` — its own npm project: Vite + `@vitejs/plugin-react`, strict TS (rule 23 applies fully), its own `eslint.config.js`/`.prettierrc` (tabs), `vitest.config.ts` (jsdom), `web/.env.example` with `VITE_API_URL=http://localhost:3000` (rule 27), scripts: `dev`, `build`, `test:web`, `verify:web` (= typecheck + lint + tests), `lint:web`, `format:web`, `generate:api`.
- Dependencies EXACTLY per §11 web column (rule 8): runtime `react`, `react-dom`, `axios`, `react-hook-form`, `i18next`, `react-i18next`, `@fontsource/roboto`; dev `vite`, `@vitejs/plugin-react`, `typescript`, `@types/react`, `@types/react-dom`, `vitest`, `@testing-library/react`, `@testing-library/user-event`, `jsdom`, `sass`, `orval`, `prettier`, `eslint`, `@eslint/js`, `typescript-eslint`, `eslint-config-prettier`.
- `web/src/styles/` — `_tokens.scss` (EVERY value from `docs/figma/style`, named by meaning: `$color-card-bg: #522473`, `$color-page-bg: #EAECEF`, …), `_mixins.scss` (`@mixin mobile` at ≤ 919 px), the reset; components style EXCLUSIVELY through CSS Modules drawing from the tokens — a hardcoded hex/px in a component is a review failure (the figma-to-scss DoD grep).

### C. The contract client and the boot (proposal §10)

- **orval** generates the axios client from `../api/openapi.json` into `web/src/api/generated/` — COMMITTED (the CI drift guard regenerates and diffs); the HTTP layer is a custom axios mutator: the base URL resolved as **`import.meta.env.VITE_API_URL ?? ''` — the fallback is EXPLICIT and load-bearing** (the human's review finding: in the CI `vite build` no `.env` exists, the env is `undefined`, and Vite reports nothing — without the `??` the runtime would construct `undefined/api/…` or an undefined axios base; the empty string IS the §10 production semantics: same-origin relative calls through the 0.10.0 Router); plus a client-side timeout and the error mapping onto `ErrorCode`/`key` — components NEVER see a raw `AxiosError` and NEVER call the generated client directly; typed hooks (`useConvert`, …) sit above.
- **`web/orval.config.ts` is the generator's contract** (the human's review finding — without it `npm run generate:api` is just a script name): `input: '../api/openapi.json'`; `output.target: 'src/api/generated/client.ts'` with `output.schemas: 'src/api/generated/model'`; `output.client: 'axios-functions'` (plain typed functions — no react-query layer, the §10 hooks are hand-written); `output.override.mutator: { path: 'src/api/mutator.ts', name: 'apiInstance' }` so EVERY generated call flows through the custom axios layer; `output.prettier: true` so the committed output matches the repo format and the drift guard never trips on formatting.
- **The boot:** a full-page spinner until `GET /api/init` AND `GET /api/currencies` resolve; an init failure → the single hardcoded fallback message; a currencies failure → an error state with a retry. The texts and the currencies live in React Context for the whole session. The request state everywhere is the §10 discriminated union (`idle | loading | success | error`).
- **i18n:** i18next + react-i18next initialized from the downloaded translations; the language pick: localStorage `language` → `navigator.languages` ∩ the server list → `en`; `<html lang>` follows. The no-fallback configuration per the analysis. (The visible changer arrives in 0.10.0 — the mechanism ships now.)

### D. The converter (proposal §10; the Figma export)

- The layout per Figma: the centered H1, the purple form card (Amount / From / To — three fields in one row on web, stacked on mobile via the grid template), the Convert button, the Result card (the label + the `formatMoney` value; space reserved — CLS ≈ 0). Pixel-perfect against `docs/figma/style` at BOTH breakpoints.
- The form: react-hook-form; Convert is an EXPLICIT action (no live recompute — §10: typing "100" must not record three conversions); the button disabled while invalid or in flight (double-submit prevention — the FE never retries a POST, §6); autofocus on the amount; Enter submits.
- **The selects are the §10 ARIA combobox** (own implementation, no library): typing filters, ↑/↓ navigate, Enter selects, Esc/outside-click closes, `role="combobox"` + `aria-expanded` + `aria-activedescendant`; the value renders code + name; **the same currency cannot be picked on both sides** (each select excludes the other's value — the backend 400 stays as defense in depth).
- **The helpers are the ONLY formatting place:** `formatMoney`/`formatCount` as a **custom deterministic implementation — NOT `Intl.NumberFormat`** (the human's review finding; §10 amended per rule 9): ICU would not even produce the design string — the `cs-CZ` grouping separator in CLDR is U+00A0/U+202F, never the plain space Figma shows — and CLDR drift across Node releases changes separators under our feet. The format is FIXED literally: the thousands separated by a REGULAR SPACE (U+0020, pinned — the design literal), a comma before exactly 2 decimals, the currency code after the amount (`4 942,52 CZK`); built by string manipulation (the §5 discipline), independent of every locale, ICU build and runtime version. The format NEVER changes with the language; the FE never rounds (§5 — the API result arrives rounded).

### E. CI and the drift guards (proposal §7)

- The web job in `ci.yml`: `npm ci` → `verify:web` → `vite build` (build errors die in the PR, not at deploy) → the generated-client drift guard (`npm run generate:api` + `git diff --exit-code web/src/api/generated`); runs in parallel with the api job; no dynamodb service needed.

### Closing the version (rules 14, 15 — in this order)

1. The version 0.9.0 in `api/package.json` FIRST (the single source of truth — rule 15); the README badge/status follow.
2. `api`: `openapi.json` re-exported AFTER the bump (the CORS plugin adds no contract change — the version line only expected).
3. `CHANGELOG.md` — the `[0.9.0] — YYYY-MM-DD HH:MM` entry.
