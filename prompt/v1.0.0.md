## Solution design

### A. The future vision (rule 17 — the section right after the badges)

- A `## Future vision` section at the top of the README: the assignment's reflection question answered in first person, built from the §12/diary-recorded core (the architecture-time argument + the verification argument), a short paragraph — not an essay.
- Drafted by the AI from the recorded decision, approved or edited by the human BEFORE the commit (rule 9).

### B. The time budget

- Extract all `[YYYY-MM-DD HH:MM]` datetimes from CHANGELOG.md and AI_DIARY.md; cluster into work blocks (a gap > 60 minutes starts a new block); sum the spans; the table and the resulting total go into the answer for the human's confirmation.
- The confirmed number lands in the README (one honest line with the method — "summed from the datetimed changelog/diary records") — placed in the AI collaboration section, where the process artifacts already live.

### C. The completeness pass

- The diary: the final read-through (rule 21, the template); the v1.0.0 record is the closing entry.
- The README: the rule 17 structure audit; the badge/status/`/health` example to 1.0.0; Project Status — every row ✅ including 1.0.0; the live URLs spot-checked; the Author & License section made true per the license decision (open question 1).
- The submission checklist in the answer: each assignment deliverable → its artifact and location.

### D. The production redeploy (the human; deny-listed)

- The version bump changes `/health` — after the merge the human redeploys so production reports 1.0.0; the spot check is one `GET /health`.

### Closing the version (rules 14, 15 — in this order)

1. The version 1.0.0 in `api/package.json` FIRST; the README badge/status follow.
2. `api`: `openapi.json` re-exported AFTER the bump (the version line is the only expected change); the web client regenerated (the drift guards stay green).
3. `CHANGELOG.md` — the `[1.0.0] — YYYY-MM-DD HH:MM` entry; the GitHub release note = the entry (rule 16).
